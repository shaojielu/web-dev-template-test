Web Dev Template - 全面代码审查报告

项目概览

FastAPI (Python 3.13) + Next.js 16 (React 19) 全栈 Web 应用模板，含 PostgreSQL、Docker
Compose、Traefik 反向代理，以及完整的 CI/CD 流水线。

---

一、系统架构设计

评价：良好 (8/10)

优点：

- 清晰的前后端分离架构，后端 API 与前端 SSR 各自独立部署
- 后端采用经典的分层架构：routes → services → models，职责清晰
- 依赖注入通过 FastAPI Depends 实现，SessionDep、CurrentUserDep、CurrentActiveUserDep 封装合理
- Docker Compose + Traefik 的组合实现了开发/生产环境的统一，且支持 Docker Swarm 水平扩展
- 前端正确使用了 Next.js App Router 的 Server Components + Server Actions 模式

问题与建议：

1.  缺少 Next.js Middleware 层 — 路由保护完全依赖 apiFetch() 调用时的 401
    自动跳转。这意味着如果一个受保护页面没有调用 API（纯静态展示），它不会被保护。建议添加 middleware.ts
    检查 cookie 存在性：
    // middleware.ts
    export function middleware(request: NextRequest) {
    const token = request.cookies.get('access_token');
    if (!token && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url));
    }
    }
2.  缺少 API 版本化策略文档 — 有 /api/v1 前缀，但没有关于如何迁移到 v2 的约定或策略。
3.  前端没有全局错误边界 — 只有 invoices/error.tsx 有错误边界，dashboard/ 和 customers/ 没有。如果 API
    异常，用户可能看到白屏。
4.  缺少请求重试机制 — 前端 apiFetch() 没有网络重试逻辑（后端 backend_pre_start.py 倒是有 tenacity
    重试）。

二、数据库设计

评价：良好 (7.5/10)

三范式合规性分析：

┌──────┬─────────┬───────────────────────────────────────────────────────────────┐
│ 范式 │ 状态 │ 说明 │
├──────┼─────────┼───────────────────────────────────────────────────────────────┤
│ 1NF │ ✅ 满足 │ 所有字段都是原子值，无重复列 │
├──────┼─────────┼───────────────────────────────────────────────────────────────┤
│ 2NF │ ✅ 满足 │ 所有非主属性完全依赖于主键（UUID） │
├──────┼─────────┼───────────────────────────────────────────────────────────────┤
│ 3NF │ ✅ 满足 │ 无传递依赖 — invoice 的 customer 信息通过 FK 关联而非冗余存储 │
└──────┴─────────┴───────────────────────────────────────────────────────────────┘

表结构（3 张表）：

users → id(PK), email(UNIQUE), full_name, hashed_password, is_active, is_superuser
customers → id(PK), email(UNIQUE), name, image_url
invoices → id(PK), customer_id(FK→customers), amount, status, date

优点：

- UUID 主键避免了顺序暴露和分布式冲突
- 正确使用了 Numeric(12,2) 存储金额，避免浮点精度问题
- server_default=func.now() 确保时间戳由数据库生成，不受应用时区影响
- CHECK 约束在 migration 中限制 status IN ('pending', 'paid')
- 索引策略合理：customer_id、status、date、email 都有索引

问题与建议：

1.  Invoice 的 amount 缺少数据库级正数约束 — 仅在 Pydantic schema 中验证 amount > 0，数据库层面没有
    CHECK (amount > 0) 约束。如果有其他路径写入数据（如直接 SQL 或管理脚本），可能写入负数。

- 修复文件: 新增 Alembic migration 添加 CHECK (amount > 0)

1.  updated_at 字段使用 onupdate=func.now() — 这是 SQLAlchemy ORM 级别的事件，不是数据库
    trigger。如果通过原始 SQL UPDATE 语句修改记录，updated_at 不会更新。建议改用 PostgreSQL trigger：
    CREATE OR REPLACE FUNCTION update_updated_at()
    RETURNS TRIGGER AS $$
    BEGIN NEW.updated_at = NOW(); RETURN NEW; END;

$$
LANGUAGE plpgsql;
3. 缺少软删除机制 —
所有删除都是硬删除（session.delete()）。对于发票等业务数据，生产环境通常需要软删除（is_deleted
字段）以满足审计需求。
4. Customer 与 User 之间没有关联 — 当前设计中 customer 是全局数据，任何认证用户都能看到所有
customer/invoice。如果是多租户场景，需要添加 owner_id 或租户字段。
5. 缺少数据库级并发控制 — 没有乐观锁（如 version 字段）或 SELECT FOR UPDATE。在高并发更新同一 invoice
 时可能出现数据竞争。

三、后端设计与实现

 评价：优秀 (8.5/10)

 优点：
- 全异步栈：asyncpg + AsyncSession + async def 路由，性能优秀
- 严格 mypy 类型检查，代码类型安全性高
- 分层清晰：路由层薄（仅接参数、调用服务、返回响应），业务逻辑在 service 层
- Session 生命周期管理正确：成功自动 commit，异常自动 rollback
- 金额处理使用 Decimal.quantize(ROUND_HALF_UP) 确保精度
- 密码恢复端点防用户枚举攻击（始终返回 200）
- 安全头完备：nosniff、DENY、HSTS（仅生产）、Referrer-Policy
- Sentry 集成，带 request ID 追踪

 问题与建议：

 1. 缺少速率限制 — 登录端点 (/login/access-token)、密码恢复端点没有速率限制，容易遭受暴力破解。
  - 建议: 使用 slowapi 或自定义中间件限制，如每 IP 每分钟 5 次登录尝试
2. JWT 无法撤销 — 没有 token 黑名单机制。如果用户修改密码或被禁用，已发出的 token
在过期前（2小时）仍然有效。
  - 建议: 至少在 get_current_user() 中检查 user.is_active（当前在 get_current_active_user
检查，但非所有路由都用这个依赖）
  - 实际查看: get_current_user 只验证 token 和查找用户，get_current_active_user 才检查
is_active。路由中用 CurrentActiveUserDep 的地方是安全的，但如果有人只用
CurrentUserDep，被禁用的用户仍可访问。
3. 搜索使用 ILIKE %query% — invoice.py:67-74 的搜索实现 f"%{query}%" 存在两个问题：
  - 没有对 % 和 _ 进行转义，用户输入这些字符会干扰搜索
  - ILIKE 前缀通配符 %query% 无法利用 B-tree 索引，大数据量时性能差。建议考虑 PostgreSQL 全文搜索或
pg_trgm 扩展
4. fetchCustomers 硬编码 limit=500 — data.ts:89 '/api/v1/customers/?skip=0&limit=500' 如果 customer
超过 500 条，创建 invoice 时的客户下拉列表会不完整。应该实现搜索或分页下拉。
5. OpenAPI 文档在生产环境暴露 — main.py:43 openapi_url=f"{settings.API_V1_STR}/openapi.json"
在所有环境都可访问。建议在生产环境禁用或添加认证。
6. CORS allow_methods 包含 PUT — 但实际 API 没有 PUT 端点（只用
PATCH），应该移除。虽然无害，但不符合最小权限原则。
7. custom_generate_unique_id 假设所有路由都有 tags — main.py:24 route.tags[0] 如果某个路由没有 tag
会抛出 IndexError。

四、前端设计与实现

 评价：良好 (7.5/10)

 优点：
- 正确使用 Server Components 进行数据获取，减少客户端 JS 体积
- Server Actions 处理表单提交，内置 CSRF 防护
- server-only 标记确保 API wrapper 不被打包到客户端
- Zod 表单验证 + ARIA live regions 提供了良好的无障碍体验
- 重定向 URL 清洗 (sanitizeRedirectTo) 防止开放重定向漏洞
- HTTP-only + SameSite Strict cookie 防止 XSS/CSRF
- 骨架屏 + Suspense 提供良好的加载体验
- 响应式设计，移动端用卡片布局，桌面端用表格

 问题与建议：

 1. isTokenExpired() 定义但未使用 — lib/auth.ts 导出了 token
过期检查函数，但没有在任何地方被调用。应该在关键位置（如 layout）使用，或者删除这段未使用代码。
2. Server Action 错误处理不够健壮 — createInvoice 和 updateInvoice 在 apiFetch 失败时会直接 throw（被
 Next.js 错误边界捕获），但 deleteInvoice 没有 try-catch，且没有向用户展示删除失败的反馈。
3. fetchInvoicesPages 效率低 — data.ts:55-68 为了获取总页数，发起了一个 limit=1 的请求来获取
count。这意味着每次页面加载要发两个请求（一个获取数据，一个获取总数）。应该从 fetchFilteredInvoices
的响应中提取 count。
4. 前端 .env 中有无用的 Vite 配置 — VITE_API_URL 是 Vite 的环境变量格式，在 Next.js
中不起作用，应该清理。
5. CSS 变量定义了暗色模式但未实际使用 — globals.css 中定义了 @media (prefers-color-scheme: dark) 的
CSS 变量，但 UI 组件全部使用硬编码的 Tailwind 类名（如
bg-white），暗色模式实际上不生效。要么实现完整的暗色模式，要么移除这些无用的 CSS 变量。
6. Revenue Chart 是纯 CSS 实现 — 柱状图用 CSS grid 手动绘制，没有使用图表库。虽然轻量，但 Y
轴标签是硬编码的 $0 - $100K，不会根据实际数据动态调整。如果收入超过 100K，图表展示会失真。
7. 客户头像使用外部服务 — i.pravatar.cc
是第三方头像服务，生产环境依赖外部服务不够可靠。建议使用本地生成的头像（如首字母
avatar）或将图片存储在自己的 CDN。

 安全风险与建议：

 1. [高] 缺少速率限制 —
登录、密码恢复、注册端点没有速率限制。攻击者可以无限制尝试暴力破解。这是投入生产前必须解决的问题。
2. [中] JWT 使用 HS256 而非 RS256 — security.py:9 使用对称签名算法。如果 SECRET_KEY
泄露，攻击者可以伪造任意 token。RS256 (非对称) 允许只暴露公钥用于验证，更安全。对于单服务架构 HS256
可以接受，但如果有多个服务验证 token，应该改用 RS256。
3. [中] 无 Token Refresh 机制 — 用户每 2 小时必须重新登录。没有 refresh token
意味着无法实现"记住我"功能，也无法在不重新登录的情况下续期。
4. [中] X-Request-ID 接受客户端输入 — main.py:89 直接使用请求头中的
X-Request-ID。恶意用户可以注入超长字符串或特殊字符到日志中（日志注入攻击）。应该验证格式（如 UUID
格式）。
5. [低] Cookie 的 secure 标志依赖 x-forwarded-proto — actions.ts:78 通过 x-forwarded-proto
判断是否设置 secure cookie。如果反向代理配置错误导致此头缺失，cookie 在 HTTPS 下也不会标记为 secure。
6. [低] bcrypt.gensalt() 使用默认 rounds=12 — 这对 2026 年来说可以接受，但随着硬件进步，建议显式设置
rounds 并在配置中可调。

七、DevOps 与基础设施

 评价：优秀 (9/10)

 优点：
- Docker 多阶段构建，镜像体积优化
- 完整的 CI/CD 流水线：lint → test → build → deploy
- GitHub Actions 支持路径过滤（只有相关文件变更才触发）
- Playwright E2E 测试支持 4 shard 并行
- Dependabot 自动依赖更新
- Docker Swarm 部署支持滚动更新 + 自动回滚
- Traefik 自动 HTTPS (Let's Encrypt)
- 健康检查覆盖所有服务

 问题与建议：

 1. 缺少数据库备份策略 — compose 中没有数据库备份相关配置。生产环境必须有定期备份。
2. 缺少监控/告警 — 没有 Prometheus/Grafana 或类似的指标收集。Sentry 只覆盖错误，不覆盖性能指标。
3. 缺少日志聚合 — 各容器日志独立，没有 ELK/Loki 等集中日志方案。
4. compose.override.yml 的 Traefik 开启了 insecure API — 开发环境可以，但要确保不会意外用于生产。
5. 后端测试覆盖率 90% 要求 — CI 中设置了 --fail-under=90，这很好，但前端没有单元测试覆盖率要求。

12. 添加前端组件单元测试
13. 添加数据库备份方案到 compose
